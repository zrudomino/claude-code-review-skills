## Key Decisions Made

- **Decision**: Atomic sentinel lock replaces PID+heartbeat
  - **Context**: Claude cannot check PIDs, run background timers, or access process.platform
  - **Rationale**: Timestamp-based staleness (written_at > 15min AND updated_at > 15min) is the only lock mechanism Claude can actually implement. 15-minute threshold chosen as longer than typical stage durations, with acknowledged trade-off during long agent waits.

- **Decision**: Convergence check evaluates prior run data only, not "current findings"
  - **Context**: Convergence runs before Stage 0, so no current findings exist yet
  - **Rationale**: Stop conditions 1-5 can all be evaluated from the prior run's findings and run_history. Actual finding-to-finding matching between new and old findings deferred to post-Stage-2.

- **Decision**: Gate 4 (API fuzz/schemathesis) made advisory-only everywhere
  - **Context**: Gate 4 was simultaneously listed as required, advisory, and immediate-escalation across different files -- a triple contradiction
  - **Rationale**: Schemathesis results are too noisy for hard-fail. Advisory warns but never blocks. Consistent across SKILL.md, both gate-registry files, and retry-protocol.

- **Decision**: Sequential batch processing with cap of 8, one worktree at a time
  - **Context**: Original design had 15 concurrent worktrees which exceeded Claude's tracking capacity
  - **Rationale**: Sequential processing is simpler, debuggable, and prevents the worktree explosion problem. Cap of 8 prevents runaway sessions. Cherry-pick after each success enables regression detection.

- **Decision**: Foreground agents for verification after background agents stalled
  - **Context**: Background verification agents ran 35+ minutes with no output (output files only populate on completion)
  - **Rationale**: Switched to foreground Skeptic + Opus agents to get results directly. Background agents eventually completed but the foreground results were already processed.

- **Decision**: `fixed_by_run` references the review-pipeline run_id, not a fix-and-verify run_id
  - **Context**: Wave 3 found that fix-and-verify has no run_id generation step
  - **Rationale**: The value written should be the run_id from the state file (the review-pipeline run that created the finding), not a nonexistent fix-and-verify run_id.

## Pitfalls & Gotchas

- Background agents take very long for thorough reviews (10 files, adversarial analysis) -- 35+ minutes with no visible progress. Output files only populate on completion, not during execution.
- The gate-registry convenience copy in fix-and-verify had completely different timeout values from the canonical (5min vs 60s, 15min vs 300s). Easy to miss because both files were written by separate agents.
- `updated_at` was never actually written anywhere in the original rewrite despite being critical to lock staleness detection. Multiple reviewers caught this independently.
- `run_history` append was specified in convergence.md but never triggered from SKILL.md -- a subtle disconnect between reference files and the main workflow.
- The convergence "current findings" problem was found by all 4 verification agents independently -- high-confidence finding that the pre-Stage-0 placement was fundamentally flawed for fresh re-invocations.
